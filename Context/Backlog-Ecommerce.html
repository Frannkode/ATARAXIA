1Setup del proyecto + Catálogo + Ficha de producto

Dejar el proyecto corriendo en ambos entornos (dev/staging) con el modelo de datos base y un catálogo navegable (listado, filtro, buscador y ficha expandible).

Definición de Listo (DoR)
  • Logo del cliente en formato vectorial (SVG/AI) y variantes en PNG
  • Paleta de colores exacta (HEX) de lila/negro/gris — se aplica desde este sprint, no al final
  • Al menos 10 productos de muestra con fotos reales (mín. 3 fotos por producto) y descripciones
  • Definición de categorías iniciales (nombres y jerarquía si aplica)
  • Acceso a repositorio Git (GitHub) creado
  • Cuenta de Cloudinary creada (o credenciales para crearla)
  • Definición de qué campos tiene un producto (talle, color, SKU, etc.)
  • Acceso a cuenta de MercadoPago del cliente (modo test) para el spike de la Historia 1.7
1.1 — Setup del proyecto Backend / DevOps

Historia: Como equipo de desarrollo, quiero tener el monorepo/proyecto configurado con linting, CI básico y entornos, para poder empezar a construir features sin fricción.

Tareas técnicas

  • Inicializar proyecto Next.js 15 (App Router) en TypeScript, con estructura de Route Handlers para la API interna
  • Configurar Drizzle ORM con conexión a PostgreSQL serverless (Neon) y pooling de conexiones; migraciones iniciales con drizzle-kit
  • Configurar Tailwind CSS v4 + shadcn/ui con los tokens de color de marca (lila/negro/gris) definitivos desde el arranque, no placeholders
  • Configurar variables de entorno (.env.example) y documentar en README
  • Setup de Cloudinary SDK para subida de imágenes

Dependencias: Ninguna

Casos borde: Falta de credenciales de DB al desplegar → el build de CI debe fallar explícitamente con mensaje claro, no silenciosamente

Criterio de aceptación: El proyecto responde en /api/health y renderiza una página placeholder, corriendo contra la base de datos de staging en Neon.

1.2 — Modelo de datos base Backend

Historia: Como backend, quiero modelar Producto, Categoría e Imagen en Drizzle, para tener la base de datos que sustenta todo el catálogo.

Tareas técnicas

  • Definir schema Drizzle: product (nombre, descripción, SKU, precioMinorista, precioMayorista, cantidadMinimaMayorista, stock, categoriaId, activo), category, productImage

Criterio de aceptación: La API devuelve productos seedeados con su categoría e imágenes asociadas.

4h2 SP

Historia: Como visitante, quiero ver el listado de productos paginado, para poder explorar el catálogo sin que la página se sobrecargue.

Tareas técnicas

5h5 SP

4h3 SP
1.5 — Buscador de productos Frontend + Backend

Historia: Como visitante, quiero buscar productos por nombre, para encontrar un artículo específico sin navegar por categorías.

Tareas técnicas

Dependencias: Depende de: Listado de productos (Historia 1.3)

Casos borde: Búsqueda sin resultados → mensaje claro; búsqueda con caracteres especiales (SQL injection) → sanitizada por Drizzle (parametrizado) y validada con Zod

Casos borde: Producto con una sola imagen → galería debe funcionar igual sin errores; producto inactivo/eliminado accedido por link directo → página 404 amigable

Criterio de aceptación: Al entrar a un producto se ve la galería completa navegable y toda su información, incluso desde un link directo.

2Precios mayorista/minorista + Carrito + Checkout sin registro

Permitir que un visitante arme un pedido con precios diferenciados según cantidad y complete un checkout sin necesidad de crear cuenta.

Definición de Listo (DoR)
  • Reglas de negocio de mayorista: cantidad mínima para precio mayorista, si aplica por producto o por pedido total
  • Campos de envío requeridos definidos (dirección, localidad, CP, teléfono; retiro en local si aplica)
  • Definición de si el envío tiene costo fijo, variable o se coordina aparte
2.1 — Modelo de precios mayorista/minorista Backend

Historia: Como backend, quiero que cada producto tenga precio minorista y mayorista con una cantidad mínima de activación, para soportar ambos tipos de venta desde el mismo catálogo.

Tareas técnicas

  • Extender schema de product: precioMayorista, cantidadMinimaMayorista
  • Lógica de cálculo de precio unitario según cantidad en el Route Handler (fuente de verdad, no confiar en el frontend)
  • Endpoint que recalcula precio total dado productId + cantidad

Dependencias: Depende de: Modelo de datos base — Sprint 1

Casos borde: Producto sin precio mayorista → se usa minorista; cantidad exactamente igual al mínimo → aplica mayorista (regla ">=")

Criterio de aceptación: Con cantidad ≥ mínimo mayorista, el backend devuelve el precio mayorista correcto.

5h5 SP
2.2 — Cálculo de precio dinámico en frontend Frontend

Historia: Como comprador, quiero ver cómo cambia el precio a medida que modifico la cantidad, para entender cuándo conviene comprar mayorista.

Tareas técnicas

  • Selector de cantidad con indicador visual ("A partir de X unidades, precio mayorista")
  • Consumir el endpoint de cálculo de precio del backend

Dependencias: Depende de: Modelo de precios (Historia 2.1)

Casos borde: Cantidad 0 o negativa → input bloqueado/validado

Criterio de aceptación: El precio mostrado siempre coincide con el calculado por el backend al agregar al carrito.

4h3 SP
2.3 — Carrito de compras Frontend

Historia: Como comprador, quiero agregar, quitar y modificar cantidades de productos en un carrito, para armar mi pedido antes de pagar.

Tareas técnicas

  • Store de carrito con Zustand persistido en localStorage (soporta guest)
  • Componente de carrito (drawer o página) con edición de cantidades y eliminación
  • Recalcular precio total (saltos mayorista/minorista) en cada cambio

Dependencias: Depende de: Cálculo de precio dinámico (Historia 2.2)

Casos borde: Producto desactivado mientras el usuario navegaba → se marca y bloquea el checkout; stock insuficiente → aviso y ajuste sugerido

Criterio de aceptación: El carrito persiste entre recargas y refleja el precio correcto ante cualquier cambio de cantidad.

5h5 SP
2.4 — Checkout sin registro Frontend + Backend

Historia: Como comprador, quiero completar mi compra dejando solo mis datos de contacto y envío, para no tener que crear una cuenta.

Tareas técnicas

  • Formulario de checkout (nombre, email, teléfono, dirección) validado con Zod
  • Server Action / Route Handler POST /api/orders que crea el pedido en estado pendiente_pago con snapshot de productos y precios

Criterio de aceptación: Un visitante sin cuenta puede generar un pedido en estado pendiente, listo para pasar a pago.

3Integración MercadoPago + Webhooks + Notificaciones automáticas

Cobrar automáticamente vía MercadoPago y notificar al cliente el estado de su pedido sin intervención manual.

Definición de Listo (DoR)
  • Cuenta de MercadoPago del cliente creada (modo test) con Access Token y Public Key
  • Definición de qué pasa si el pago es rechazado (¿se libera stock? ¿se avisa para reintentar?)
  • Cuenta de Resend / dominio verificado para envío de emails
  • Textos definidos de los emails transaccionales (confirmación, rechazo)
3.1 — Integración Checkout Pro de MercadoPago Backend

Historia: Como comprador, quiero ser redirigido a MercadoPago para pagar mi pedido, para completar la compra de forma segura.

Tareas técnicas

  • Integrar SDK de MercadoPago, crear preferencia (items, external_reference = orderId, back_urls, notification_url)
  • Route Handler POST /api/orders/:id/checkout que genera la preferencia y devuelve el link de pago
  • Reserva de stock al generar la preferencia para evitar sobreventa

Dependencias: Depende de: Checkout sin registro — Sprint 2

Casos borde: Usuario abandona el pago → orden queda pendiente_pago con expiración configurable; reintento de pago sobre la misma orden → reutilizar/regenerar preferencia sin duplicar la orden

Criterio de aceptación: Al confirmar el checkout, el comprador es redirigido a MercadoPago con el monto e ítems correctos.

6h5 SP
3.2 — Webhook de confirmación de pago (IPN) Backend

Historia: Como sistema, quiero recibir la notificación de MercadoPago cuando un pago se aprueba, rechaza o queda pendiente, para actualizar el pedido automáticamente.

Tareas técnicas

  • Route Handler POST /api/webhooks/mercadopago que consulta el pago real vía API (nunca confiar solo en el payload)
  • Idempotencia: registrar paymentId procesados
  • Actualizar estado de la orden según approved / rejected / pending / in_process

Dependencias: Depende de: Integración Checkout Pro (Historia 3.1)

Casos borde: Webhook duplicado (reintentos de MP) → idempotencia por paymentId; webhook antes de que exista la orden → reintento/cola; external_reference inexistente → loggear y responder 200 igual

Criterio de aceptación: Un pago aprobado actualiza la orden en menos de 10s, sin duplicar procesamiento ante reintentos.

6h8 SP
3.3 — Actualización automática de estado de pedido Backend

Historia: Como administrador, quiero que el estado del pedido refleje automáticamente el resultado del pago, para no verificar manualmente en MercadoPago.

Tareas técnicas

  • Máquina de estados de order: pendiente_pago → pagado / rechazado / en_proceso
  • Liberar stock reservado si el pago es rechazado o expira
  • Descontar stock definitivamente cuando el pago se aprueba

Dependencias: Depende de: Webhook (Historia 3.2)

Casos borde: Pago aprobado pero stock ya no disponible → marcar orden para revisión manual, no fallar silenciosamente

Criterio de aceptación: Toda orden pagada queda en estado pagado con el stock descontado, sin intervención manual.

4h3 SP
3.4 — Notificación automática al cliente Backend

Historia: Como comprador, quiero recibir un email cuando mi pago fue procesado, para saber si mi pedido está confirmado.

Tareas técnicas

  • Integrar Resend + React Email para envío y diseño de emails transaccionales
  • Templates de email (componentes React): pago aprobado, pago rechazado
  • Disparar envío desde el flujo de actualización de estado (Historia 3.3)

Dependencias: Depende de: Actualización automática de estado (Historia 3.3)

Casos borde: Falla el envío de email → no debe bloquear la actualización del pedido; reintento o log para reenvío manual

Criterio de aceptación: Al aprobarse o rechazarse un pago, el comprador recibe un email en menos de 1 minuto.

5h5 SP
Testing del sprint Simulación de pagos en sandbox de MercadoPago (aprobado, rechazado, pendiente) verificando orden, stock y email. Incluye test de webhook duplicado.
Criterio: Los 3 escenarios funcionan correctamente y un webhook reenviado no duplica efectos. — 5h / 3 SP

4Panel admin: login/roles + CRUD de productos

Dar al personal autorizado un panel privado para gestionar el catálogo (precios, descripciones, categorías) sin tocar la base de datos directamente.

Definición de Listo (DoR)
  • Lista de usuarios administradores iniciales y si hay distinción de roles

4.1 — Login de administrador con roles Backend

Historia: Como administrador, quiero iniciar sesión con mi usuario y contraseña, para acceder al panel de gestión de forma segura.

Tareas técnicas

  • Configurar Better Auth con proveedor de email + contraseña para el acceso admin
  • Definir tabla de usuarios admin y rol por usuario en Drizzle, integrada al esquema de Better Auth
  • Configurar expiración/rotación de sesión y flujo de logout

Dependencias: Ninguna

Casos borde: Intentos fallidos repetidos → rate limiting mediante el plugin de Better Auth; sesión expirada → redirigir a login, no panel roto

Criterio de aceptación: Un admin válido se loguea y su sesión persiste vía cookie segura; credenciales incorrectas son rechazadas con mensaje genérico.

5h5 SP
4.2 — Middleware de protección de rutas admin Backend

Historia: Como sistema, quiero que ninguna ruta del panel ni de la API admin sea accesible sin una sesión válida, para proteger la información sensible del negocio.

Tareas técnicas

  • Middleware de Next.js que valida la sesión de Better Auth en todas las rutas /admin y /api/admin/*
  • Redirección a login si no hay sesión válida
  • Verificación de rol para acciones sensibles vía el plugin de roles de Better Auth

Dependencias: Depende de: Login de administrador (Historia 4.1)

Casos borde: Acceso directo por URL sin sesión → redirección a login, no error 500; rol insuficiente → 403 explícito

Criterio de aceptación: Ninguna ruta ni Route Handler admin responde datos sin una sesión válida.

3h2 SP

Tareas técnicas

Tareas técnicas

5Panel admin: gestión de pedidos, envíos y stock

Que el personal pueda operar el día a día de pedidos (ver, actualizar envío, corregir estado) y mantener el stock bajo control.

Definición de Listo (DoR)
  • Estados de envío definidos (ej. "preparando", "despachado", "entregado")
  • Definición de si el stock es por producto simple o por variante (talle/color)
5.1 — Listado y detalle de pedidos Frontend + Backend

Historia: Como administrador, quiero ver todos los pedidos con su estado y detalle, para gestionar la operación diaria de la tienda.

Tareas técnicas

  • Route Handler GET /api/admin/orders con filtros por estado y fecha
  • Vista de detalle (productos, envío, estado de pago, estado de envío)

Dependencias: Depende de: Modelo de Order — Sprint 2; Protección de rutas — Sprint 4

Casos borde: Pedido con pago rechazado en el listado → debe distinguirse visualmente de los pagados

Criterio de aceptación: El administrador ve todos los pedidos y los filtra por estado sin acceder a la base de datos.

5h5 SP
5.2 — Actualización de estado de envío Frontend + Backend

Historia: Como administrador, quiero marcar un pedido como "despachado" o "entregado", para que el estado de envío esté siempre actualizado.

Tareas técnicas

  • Route Handler PATCH /api/admin/orders/:id/shipping-status

  • (Opcional) campo de número de seguimiento manual

Dependencias: Depende de: Listado de pedidos (Historia 5.1)

Criterio de aceptación: Cambiar el estado de envío se refleja inmediatamente en el detalle del pedido.

4h3 SP
5.3 — Actualización manual de estado de pedido Backend

Historia: Como administrador, quiero poder corregir manualmente el estado de un pedido, para resolver casos excepcionales que el sistema automático no contempla.

Tareas técnicas

  • Route Handler PATCH /api/admin/orders/:id/status con override manual
  • Registro de auditoría (quién cambió el estado y cuándo)
